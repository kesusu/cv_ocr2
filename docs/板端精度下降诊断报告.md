## 二、排查过程与发现

### 发现 A: 两个 INT8 模型的区别 (非根因但重要)

| | `ch_PP-OCRv4_rec_mobile_int8.onnx` (官方) | `ch_PP-OCRv4_rec_mobile_custom_int8.onnx` (自量化) |
|:-:|:--|:--|
| 来源 | PaddleOCR 官方模型库下载 | 项目用 ort.quantize_static 自制 |
| 校准数据 | **通用数据集** (非本项目图片) | photos/ 下3张药品说明书 |
| PC端精度 | 未单独测试 | 余弦相似度 0.97 (校准集太少) |
| PC端速度 | 快47% | **慢4.7x** (QDQ反量化开销) |
| 板端精度 | **❌ 崩坏: 58%空串+乱码** | 未测试 |

**结论**: 这两个 INT8 模型都有问题, 不应用于生产。生产继续用 FP32。

> 详细对比见 `DEPLOY.md` 末尾 "INT8 量化模型说明" 章节。

### 发现 B: ORT 版本差异 (Docker vs PC)

```
PC 端:    onnxruntime == 1.19.2
Docker:    onnxruntime == 1.23.2
开发板:    (待确认, 可能也是 1.20+)

版本差异可能影响:
  - quantize_static API 参数名变化 (calibration_dataset → calibration_data_reader)
  - QDQ 执行行为差异
  - 默认 execution provider 行为
```
**结论**: 版本差异存在但不是主因。两个版本的 FP32 推理结果应该一致。

### 发现 C: ⭐⭐⭐ 真正的根因: dict_size 与模型输出维度不匹配 ⭐⭐⭐

这是 **ocr_board.py 板端模式的一个 bug**, 导致所有通过 SDK OnnxContext 的 Rec 输出全部错位。

#### 问题代码位置

文件: `ocr_board.py` 第 730-743 行 `_run_rec()` 方法:

```python
def _run_rec(self, input_data):
    """执行 Rec 推理"""
    input_feed = {'x': input_data.astype(np.float32)}
    output_names = ['softmax_11.tmp_0']
    
    if self._is_pc_mode:
        # ===== PC 模式 (正常工作) =====
        out = self.rec_session.run(output_names, input_feed)
        return out[0]  # numpy ndarray, shape=(B, T, C), C=6625
    
    else:
        # ===== 板端模式 (有 BUG!) =====
        result = self.rec_session.Execute(output_names, input_feed)
        val = result.get('softmax_11.tmp_0', [])   # flat list
        batch = input_data.shape[0]
        
        # ❌❌❌ 这里是问题所在 ❌❌❌
        seq_len = len(val) // batch // self.dict_size   # self.dict_size = 6622
        return np.array(val).reshape(batch, seq_len, self.dict_size)  # reshape 成 (B,T,6622)
```

#### 数据流分析

```
步骤 1: 字典加载 (L133-137 load_dict())
  ppocr_keys_v1.txt 内容格式:
    第1行: 空行 (代表 CTC blank token)
    第2~6622行: 6621个可见字符
  → self.dict_size = len(chars) = 6622 ✓ (字典本身没问题)

步骤 2: Rec 模型推理
  ch_PP-OCRv4_rec_mobile.onnx 实际输出:
    shape = (batch, time_steps, 6625)   ← 注意! 最后维是 6625 不是 6622!
    
  为什么多出 3?
    PP-OCRv4 的 CTC 字典结构:
      idx=0:     <blank>   (CTC blank token)
      idx=1~N:   可见字符   (来自 ppocr_keys_v1.txt)
      idx=N+1:   <sos>      (start of sequence, 部分版本)
      idx=N+2:   <eos>      (end of sequence, 部分版本)  
      或其他特殊token
    
  实际上: 6625 - 6622 = 3 个额外token

步骤 3: PC 模式 (正确)
  ORT run() 返回 numpy array, 自动携带 shape=(B, T, 6625)
  CTC 解码时遍历 6625 维, 正确找到每个字符的概率
  → 结果正确 ✅

步骤 4: 板端模式 (错误!)
  SDK OnnxContext.Execute() 返回 flat list, 长度 = B × T × 6625
  
  计算 seq_len:
    seq_len = (B × T × 6625) ÷ B ÷ 6622
             = T × 6625 / 6622
             = T × 1.000453...
             
  整数除法截断后 seq_len 可能偏差 ±1 或更多
  
  然后 reshape(batch, wrong_seq_len, 6622):
    总元素 = B × wrong_T × 6622  ≠  B × T × 6625 (原始元素数)
    → 要么报错(ValueError) 要么静默截断/填充!
    
  即使侥幸 reshape 成功, 维度含义也错了:
    原始 [b, t, c] 对应字符 dict_chars[c]
    错位后 [b, t', c'] 中 c' 的概率实际属于另一个字符
    → CTC 解码输出乱码/空串 ❌
```

#### 图解

```
模型真实输出 (每行是一个时间步的概率分布):
  
  step 0: [blank=0.502, '你'=0.023, '好'=0.010, ..., 'z'=0.001, <eos>=0.000]  ← 6625维
  step 1: [blank=0.945, '我'=0.002, ...]                                            ← 6625维
  ...

板端错误 reshape 后 (截断到6622维):

  step 0: [blank=0.502, '你'=0.023, '好'=0.010, ..., ???=???]                    ← 只有前6622维
                                                    ↑
                                              这里本应是<eos>的位置
                                              但现在被当成下一个step的数据了!
                                              
  或者更糟 — 整个数组被重新切分, 导致:
  step 0' 包含了 step 0 的前半部分 + step 1 的部分数据 → 完全是垃圾概率
```

## 三、解决方案

### 方案 1 (推荐): 用模型实际的输出维度 (最简单)

```python
# ocr_board.py _run_rec() 方法修改

def _run_rec(self, input_data):
    """执行 Rec 推理"""
    input_feed = {'x': input_data.astype(np.float32)}
    output_names = ['softmax_11.tmp_0']
    
    if self._is_pc_mode:
        out = self.rec_session.run(output_names, input_feed)
        return out[0]
    else:
        result = self.rec_session.Execute(output_names, input_feed)
        val = result.get('softmax_11.tmp_0', [])
        batch = input_data.shape[0]
        
        # ★ 修复: 使用模型实际输出维度 6625, 而不是字典大小 6622
        REC_OUTPUT_DIM = 6625   # PP-OCRv4 rec mobile 模型的固定输出维度
        
        seq_len = len(val) // batch // REC_OUTPUT_DIM
        raw = np.array(val).reshape(batch, seq_len, REC_OUTPUT_DIM)
        
        # ★ 只取前 dict_size 维给 CTC 解码器 (忽略额外的特殊token)
        return raw[:, :, :self.dict_size]   # (B, T, 6622)
```

**改动量**: 3 行  
**原理**: 先按正确的总维度 reshape, 再裁剪到字典大小, 保证概率分布对齐。

## 五、关于 INT8 量化的补充说明

### 为什么 QDQ 量化在 CPU 上反而变慢?

```
QDQ (Quantize-Dequantize) 格式的工作方式:

  原始 FP32 计算图:
    Input → Conv → BN → ReLU → Conv → Output
    
  插入 QDQ 节点后:
    Input → Quantize(INT8) → Dequantize(FP32) → Conv(FP32) → ... → Output
              ↑                ↑
            类型转换          类型转换
            
  在 GPU/NPU 上:
    硬件原生支持 INT8 矩阵运算 → Quantize后的数据直接用INT8算 → 快!
    
  在普通 CPU 上 (无 AVX-VNNI/ARM DotProd 指令优化时):
    Dequantize 回 FP32 → 还是用 FP32 算 → 多了两次类型转换开销 → 更慢!
```

### 为什么官方 INT8 模型在板端崩坏了?

```
PaddlePaddle 导出 INT8 ONNX 时使用的校准流程:
  
  通用数据集 (ImageNet/COCO/LMDB 等) → 统计各层激活值的范围 → 确定 scale/zero_point → 量化权重
  
  问题:
    通用数据集的像素分布 ≠ 你的药品说明书图片的像素分布
    → scale/zero_point 不适合你的输入
    → CTC 层输入被错误缩放
    → softmax 输出的概率分布被破坏
    → 高概率位置不再是正确的字符
    → CTC 解码输出空白/乱码
    
  类比: 
    用"英语考试"的标准答案去批改"中文作文", 当然全错
```

### 正确的 INT8 量化做法 (如果将来需要)

```
需要三样东西:
  1. 项目实际要识别的图片 (至少50张不同角度/光线的)
  2. PC 端的量化工具链 (onnxruntime.quantization)
  3. 目标硬件的原生 INT8 支持 (DSP/NPU/GPU)

流程:
  项目图片 → Det裁剪 → 收集text crops → 作为校准集 → quantize_static()
                                                          ↓
                                              生成专用 INT8 模型
                                                          ↓
                                              部署到支持INT8的硬件
  
注意: 如果目标还是 CPU (ORT CPU Execution Provider), 
      即使自定义校准也不会加速, 只是精度保持。
      真正加速需要 DSP (SNPE DLC) 或 NPU 或 GPU TensorRT。
      
      但本项目 SNPE/DSP 全路线已证实不可用:
        Det DLC → 固定输出shape, 精度崩
        Cls DLC → 系统性误旋转
        Rec DLC → MobileOne含5D Transpose无法转换
```

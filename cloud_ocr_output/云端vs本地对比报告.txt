六、网络影响分析 — API请求时间到底花在哪？
================================================================================

【Request时间 = 三部分之和】
  ┌─────────────────────────────────────────────────────────┐
  │  requests.post() 耗时 = T_上传 + T_推理 + T_下载        │
  │                                                         │
  │  T_上传 = 图片base64数据上传到服务器                      │
  │         ≈ 数据量 / 上行带宽  (你的图片~2MB base64)       │
  │                                                         │
  │  T_推理 = 服务端模型实际做OCR的时间                       │
  │         ≈ 固定值，跟图片复杂度有关，跟你的网速无关         │
  │                                                         │
  │  T_下载 = 结果JSON返回                                   │
  │         ≈ 很小可忽略                                    │
  └─────────────────────────────────────────────────────────┘

【旧网 vs 新网 对比验证】

  photo_4: 5.022s → 1.975s (减少3.05s)
  photo_6: 4.717s → 0.986s (减少3.73s)
  photo_8: 4.954s → 1.109s (减少3.84s)
  photo_10:5.046s → 1.151s (减少3.90s)
  photo_12:1.787s → 0.824s (减少0.96s)

  ★ 所有图片都减少了约 3~4s → 这就是网络传输节省的时间
  ★ photo_12 原本就快(1.8s)，说明那张图的服务端推理本身就短
  ★ 新网下最低 0.82s → 接近纯推理时间（上传+下载已很快）

【结论：能不能"提前请求"？】

  短答案：**不能预请求，但可以优化网络部分**

  详细解释：
  1. 每次请求必须带图片数据（~2MB base64），无法预先缓存
  2. 服务端推理时间固定，无法绕过（这是核心计算）
  3. 能优化的只有"网络传输"部分：

     ✅ 可做的优化：
     - HTTP Keep-Alive 复用连接（避免每次TCP握手 ~50-100ms）
     - 用 requests.Session() 替代每次新建连接
     - 压缩图片再上传（缩小base64体积）
     - 并发多张图片同时请求
     - CDN加速/就近接入点

     ❌ 无法做的：
     - "预请求"空结果（没有图片数据服务端不处理）
     - 减少服务端推理时间（那是百度的事）

  4. 你的新网络已经把传输压到很低了(0.8~2.0s)，
     再优化空间有限。剩余的 ~0.8s 基本就是服务端推理底线。

五、结论与建议
================================================================================

【选云端OCR如果】
  ✅ 优先考虑速度 (新网下平均 2.0s vs 本地 7.3s，快 3.6倍)
  ✅ 需要更高的英文/混合文字识别准确率
  ✅ 可以接受数据上传到云端
  ✅ 不想维护本地模型文件 (~16MB)
  ✅ 图片质量参差不齐（模糊/倾斜/低分辨率）
  ✅ 网络条件好（新网实测API请求仅0.8~2.0s）

【选本地OCR如果】
  ✅ 必须离线运行 (无网络环境 / 嵌入式板端部署)
  ✅ 数据隐私要求高 (医疗信息不上传)
  ✅ 图片质量好 (清晰正拍/扫描件)
  ✅ 中文为主、不需要英文准确性
  ✅ 需要完全控制推理过程 (自定义过滤/排序逻辑)
  ✅ 网络不稳定或无网络环境

【最优策略】混合使用
  1. 默认用本地OCR (快速响应，零延迟，完全离线)
  2. 当本地置信度低于阈值 或 关键词("氨麻美敏")未匹配时，
     自动 fallback 到云端 OCR 做二次验证
  3. 新网环境下云端fallback仅 ~1-2s，几乎无感
  4. 这样兼顾了离线能力(大部分请求本地秒回)、质量兜底(云端保底)
